About

An OS, not an app.

ClarityOS was built for a single problem: model output is only as useful as the context that shaped it. The OS owns the context. The model owns the compute. The contract between them does not drift.

Origin

ClarityOS started as an internal runtime to manage long-lived operator workflows: a single human (or small team) running repeated reasoning passes across legal, institutional, operational, and personal domains. Every off-the-shelf approach had the same failure: state evaporated between sessions. The model never remembered what mattered, never knew which constraints had already been checked, and never held the operator’s prior conclusions as ground truth.

The first version of the runtime fixed that by holding state outside the model. Every subsequent version has been a refinement of that invariant.

Design principles

Determinism
Every kernel call has a stable contract. Identical inputs produce identical-shaped outputs. Random behaviour is isolated to model calls and surfaced explicitly.
The OS holds state. The model does not.
Threads, vault entries, operator state, project metadata, and the operator timeline live in ClarityOS. The model receives context on every call and returns text. Nothing else crosses the boundary.
No plugins. No skills. No manifests.
The runtime does not load third-party reasoning packages. Reasoning modes are kernel functions with stable schemas. Documentation bundles describing those modes exist, but ClarityOS never imports them at runtime.
Per-user partitioning is structural.
The encrypted vault is partitioned per user at the storage layer. Cross-user reads are not blocked at the application layer — they are unreachable.
Graceful degrade is mandatory.
Every reasoning mode has a deterministic fallback. When a provider call fails, the kernel returns a structured result the caller can rely on. No 5xx for parse failures.
One contract, three surfaces.
The same operations are reachable from phone, web, and desktop. Continuity is the runtime contract, not a feature flag.

System architecture

ClarityOS is composed of four layers:

  1. Surfaces. Phone (Expo), web (Vite + React), and desktop (Electron) render views over the same cloud state.
  2. API. A FastAPI service on Cloud Run. Session-scoped, per-user. Every domain has its own router (threads, projects, vault, timeline, regression-first, billing, founder console).
  3. Kernel. Pure-Python modules that hold the reasoning contracts: intelligence routing, the clarity pipeline, the regression-first protocol, the model router, the local model runtime, the operator timeline.
  4. Storage. An encrypted local vault partitioned per user. JSON-serializable values under namespaced keys. PBKDF2-derived keys; no shared secrets.

Invariants and safety model

ClarityOS holds itself to a small set of invariants. They are checked by tests and surfaced in readiness documents on every release pass:

  • No skill imports. The runtime never imports from documentation bundles. The boundary is enforced by a test.
  • No cross-user data. Every cross-user access returns 404 with no existence leak.
  • No silent state. Every mutation that changes user-visible state emits a timeline event. Metadata-only changes (like tag edits) intentionally do not.
  • No double-charge. Billing endpoints check status before invoking PaymentIntents. Idempotent webhook handling, signature-verified.
  • No drift between surfaces. The shape of a thread on phone is the shape of a thread on web is the shape of a thread on desktop. Helpers are 1:1 across surfaces.

Each invariant is documented in the corresponding readiness file and locked by a test. When an invariant is intentionally relaxed, the readiness file says so explicitly and the change is reviewed at the architectural layer.
