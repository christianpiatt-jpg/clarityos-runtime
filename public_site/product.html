Product

Surfaces and kernel components.

ClarityOS is a single runtime exposed through three surfaces and assembled from a small number of kernel components. Each component has a stable contract and a deterministic shape.

PHONE runtime

The phone surface is a React Native / Expo application. It connects to the cloud over HTTPS, authenticates via the operator session, and renders the same vault state the web cockpit renders. Threads, projects, the operator timeline, and the regression-first packet runner are all reachable.

The phone is read-write equal to the web cockpit. There is no feature gap by device class. Network drops degrade gracefully: the runtime caches recent reads and retries writes when the connection returns.

WEB cockpit

The web cockpit is a Vite + React application. It is designed for long-form work: assembling threads, reviewing the operator timeline, running regression chains, inspecting vault state, and managing model preferences.

All routes are auth-gated by a single session header. The cockpit holds no privileged state of its own; every panel reads from the cloud and renders deterministically from the response.

CLOUD execution layer

The cloud layer is a FastAPI service that runs on Cloud Run. It is the home of the kernel: intelligence routing, the encrypted local vault, the operator timeline, threads, projects, and every reasoning mode.

Endpoints are session-scoped. Per-user partitioning is native at the vault layer, not bolted on at the application layer. Cross-user reads are not just rejected; they are unreachable.

Interpreter

The interpreter is the kernel itself. It assembles context for the model from the operator’s vault, threads, projects, and recent timeline, dispatches through the model router, and coerces the result back into a typed shape.

The interpreter does not load skill manifests. It does not branch on whether a plugin is “active.” Reasoning modes are kernel functions with stable schemas. When a new mode is added, it ships as a kernel function with its own endpoint — not as a registry entry.

Model router

The model router selects which provider executes a given task. Resolution is deterministic and observable:

  1. Explicit override (when the caller pins a model).
  2. Founder default (a global override, if set).
  3. User preferred model (from operator state).
  4. Task default (the fallback per task class).

Five provider classes are supported: OpenAI, Anthropic, Google, xAI, and a local on-device runtime. The router itself is provider-agnostic. Adding a sixth provider does not change any caller.

Clarity pipeline

The clarity pipeline is the reasoning-stability operator. It scores text by the ratio of emotive language to institutional signal and routes the consuming model into one of three reasoning modes: stabilize, expand, or normal.

When the operator reports a problem, the pipeline can trigger the regression-first protocol — a layered diagnostic walk through the failing logic chain that refuses to propose fixes until each layer has been verified.

Local vault

The vault is encrypted at rest. Per-user keys are derived from a PBKDF2 root. Values are JSON-serializable and stored under namespaced keys: threads, projects, embeddings, regression chains, regression packets, preferences, operator state.

The vault holds the canonical state. Every other component — cockpit, phone, cloud kernel — is a view over that state. Reads are partitioned by user; cross-user access is structurally impossible.

Multi-surface continuity

A thread started on phone is the same thread on web. A chain opened from web is replayable from desktop. The operator timeline aggregates activity from every surface and presents it in one chronological feed.

Continuity is not a feature. It is the runtime contract. When a new surface is added, it inherits the contract; it does not negotiate a new one.
