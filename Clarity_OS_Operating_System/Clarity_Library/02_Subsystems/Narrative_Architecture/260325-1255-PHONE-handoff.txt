2. Meta and versioning

Inside 00_Meta:

• File: README_OS.md• Content (short):• Purpose: one paragraph describing the OS.
• Scope: list of modules (01–11).
• How to read: “Start with 01 → 02 → 04 → 07 → 08.”


• File: CHANGELOG_OS.md• Format for each entry:• YYYY-MM-DD – vX.Y – [Module] – Short description
• Example: 2026-03-22 – v1.0 – Global_OS – Initial full architecture install


• File: DEWEY_INDEX.md• List all Dewey tags you’ve assigned so far with 1–2 line descriptions.



---

3. Module pattern (use this for each numbered folder)

For each module folder (e.g., 02_Basins_&_OS), create:

• INDEX.md• Sections:• Purpose: what this module does.
• Inputs: what it depends on (e.g., physical vectors, language currents).
• Outputs: what it feeds (e.g., barometers, org_suite).
• Dewey Tag: if applicable.


• MODEL.md• Clean description of the model, no examples, just structure.

• EXAMPLES.md• Concrete cases (e.g., “China basin”, “Atlantic corridor”, “Mesoamerican loop”).

• DIAGRAM_NOTES.md• Text description of diagrams you might later draw.



You can add more files later, but this pattern keeps everything consistent.

---

5. Naming conventions

• Files: snake_case or Title_Case, but consistent.
• Versions inside files:• At top: Version: v1.0 – YYYY-MM-DD

• References:• Use relative links:• e.g., in 04_Language_Currents/INDEX.md:
See ../02_Basins_&_OS/MODEL.md for basin definitions.




---

6. Update workflow (how you “file” OS updates)

When you have an update:

1. Decide module: Which folder does it belong to?
2. Open MODEL.md or EXAMPLES.md:• Add/update the relevant section.

3. Log it in 00_Meta/CHANGELOG_OS.md:• YYYY-MM-DD – v1.1 – [Module] – Added X

4. If it changes relationships:• Add a note in INDEX.md of affected modules.
