## What the Web Surface is

The Web Surface is the typed boundary between the ClarityOS web
client (the Vite/React SPA under `web/`) and the runtime spine
(BD1–BD5). It defines the request / response envelope, the
discriminated union of actions the SPA may emit, and the stable
error envelope every handler returns. The v0.2.0 contract is the
first versioned cut — it ships the shape, not the behaviour.

When v0.2.0 routing lands (see the v0.2.0 release tagging plan in
[`README.md`](../../README.md)), every adapter on the SPA side and
every handler on the runtime side type-checks against this
contract. Two callers on different contract versions can coexist
during migration: the version segment in the namespace name
(`WebSurfaceV0_2`) is load-bearing.

## Request / Response envelope

Every Web Surface call carries a `Request` and returns a `Response`.
The envelope is intentionally minimal — it mirrors what an HTTP
adapter would hand to a Node-style handler, so the contract works
unchanged whether the consumer is a Cloud Run service, an Express
middleware, or a fetch wrapper in the SPA.

```ts
interface Request {
  path:    string;                       // route path, e.g. "/me"
  method:  string;                       // HTTP verb, uppercase
  headers: Record<string, string>;       // single-value headers
  body:    unknown;                      // already-parsed request body
}

interface Response {
  status:  number;                       // HTTP status code
  headers: Record<string, string>;
  body:    unknown;                      // serialisable response body
}
```

A non-2xx response carries an [`ErrorEnvelope`](#error-envelope) in
`body` by convention.

## SurfaceAction — discriminated union

The SPA emits `SurfaceAction` values. Each variant carries `type` as
the discriminator; consumers exhaustively `switch` on `action.type`
and TypeScript narrows the remaining fields.

```ts
type SurfaceAction =
  | { type: "noop" }
  | { type: "render"; view: string; params?: Record<string, unknown> }
  | { type: "navigate"; path: string };
```

| Variant      | Purpose                                                        |
| ------------ | -------------------------------------------------------------- |
| `noop`       | Explicit no-op. Useful as a sentinel and as a safe default in any reducer that needs one during migration. |
| `render`     | Render a named view. `params` is optional and opaque to the contract. |
| `navigate`   | Change the current route path. The SPA's router consumes this. |

Add new variants here AND in `webSurfaceV0_2.ts` before they're
emitted. The exhaustive `switch + never` pattern in the tests
catches a missed update at compile time.

### Discriminator constants

```ts
WebSurfaceV0_2.SurfaceActionType.noop      // "noop"
WebSurfaceV0_2.SurfaceActionType.render    // "render"
WebSurfaceV0_2.SurfaceActionType.navigate  // "navigate"
```

Use these instead of string literals so a typo is a compile-time
`never` error rather than a silent runtime miss.

## Error envelope

```ts
interface ErrorEnvelope {
  error:   string;                       // machine-readable code
  detail?: unknown;                      // optional diagnostic payload
}
```

* `error` is required and string-typed. Examples: `"not_implemented"`,
  `"unauthorized"`, `"validation_failed"`, `"rate_limited"`.
* `detail` is optional and opaque. Use for caller-visible diagnostics
  (field names that failed validation, retry-after hints, etc.).
* **Adding fields is allowed.** Removing or renaming `error` is a
  breaking change that requires a new contract version (a sibling
  namespace `WebSurfaceV0_3`, not a mutation here).

---

## Routing is disabled by default

The v0.2.0 contract defines the shape only. There is no handler
function exported, no router wired, and no Cloud Run deployment
configured. The contract is consumable by:

* the SPA, for type-checked construction of `SurfaceAction` payloads
  and adapter response handling, and
* any future runtime handler, for type-checked validation of incoming
  requests + construction of responses.

When the v0.2.0 routing layer lands (separate task), it will
import this module and bind a handler function to a route table —
no contract changes required.

---

## Coexistence with future versions

| Version | Namespace          | Coexistence rule |
| ------- | ------------------ | ---------------- |
| v0.2.0  | `WebSurfaceV0_2`   | This document.   |
| v0.3.0+ | `WebSurfaceV0_3` (future) | Lives in `web/src/contracts/webSurfaceV0_3.ts`. Two callers on different versions can read + write each other's envelopes via a one-way adapter, but the namespaces never alias. |

The version pin (`WebSurfaceV0_2.VERSION`) is the runtime check —
log it on every adapter boundary so telemetry can correlate a
mis-versioned call.
